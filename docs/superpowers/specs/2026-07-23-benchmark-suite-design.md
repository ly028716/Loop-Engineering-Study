# Loop Engineering 评测基准设计

## 目标

建立一个确定性的评测基准（benchmark suite），在统一场景矩阵中比较 `fixed`、
`error_aware`、`memory_aware` 和 `adaptive` 四种策略，并给出可解释的综合排行榜。

基准只依赖现有项目代码和 Python 标准库。每次结果均可由 Trace 与 Artifact 回放。

## 场景矩阵

基准固定包含以下 5 个场景：

| 场景 | 主要风险 | 目的 |
| --- | --- | --- |
| `steady_progress` | 无故障 | 衡量正常目标逼近的效率。 |
| `action_failure` | 单次动作失败 | 衡量策略恢复推进的能力。 |
| `missing_feedback` | 失败后缺少恢复信号 | 衡量策略在反馈不完整时的鲁棒性。 |
| `stalled_progress` | 动作不产生进展 | 衡量安全停止与无效步骤控制。 |
| `tight_budget` | 步骤预算很小 | 衡量在严格预算下的完成能力。 |

每个场景使用固定的初始状态、动作、评估器和停止条件。场景顺序固定，运行结果可复现。

## 运行模型

新增 `experiments/benchmark_suite.py`，提供：

- `run_benchmark(output_dir: str | Path = ".loop/runs/benchmark-suite")`；
- 每个策略与每个场景组合运行 1 次，共 20 次运行；
- 每次运行写入独立 Artifact，文件名由策略名与场景名组成；
- 结果包含场景明细、策略汇总和排行榜。

基准不修改 `loop_engineering/` 的公共接口，也不引入第三方依赖。

## 指标与评分

每条场景结果至少包含：

- `strategy`、`scenario`；
- `success`、`status`、`steps`、`final_score`；
- `failure_count`、`recovery_count`、`budget_respected`；
- `artifact_path`。

对每个策略汇总以下指标：成功率、平均成功步骤数、恢复率、预算遵守率和总分。

总分范围为 0 到 100，采用固定权重：

| 指标 | 权重 | 计算方式 |
| --- | ---: | --- |
| 成功率 | 50 | 成功场景数 / 5 × 50。 |
| 步骤效率 | 25 | 仅在成功场景上计算；以同场景成功策略中的最少步骤为基准，按比例累计后取平均。没有成功场景时为 0。 |
| 恢复能力 | 15 | 在 `action_failure` 和 `missing_feedback` 场景中，成功且出现至少 1 次恢复决策的比例 × 15。 |
| 预算遵守度 | 10 | 预算遵守场景数 / 5 × 10。 |

排行榜按 `total_score` 降序排列；总分相同时，按平均成功步骤数升序排列；仍相同时，按策略名称升序排列。

## 预期行为

`adaptive` 策略应在该确定性矩阵的综合得分上高于 `fixed`。这不是对所有未来任务的普遍结论，而是本基准设置下的可验证学习结论。

失败或停滞场景不要求成功；它们应保留停止原因、步骤数和预算遵守结果，以避免将安全停止错误地计为未观测运行。

## 验收条件

1. 基准按固定顺序运行 5 个场景和 4 个策略，产出 20 个 Artifact；
2. 每个 Artifact 均可通过 `load_run_artifact()` 加载；
3. 结果包含场景明细、策略汇总和稳定排序的排行榜；
4. 所有评分字段处于规定范围，排行榜使用规定的同分规则；
5. `adaptive` 的总分严格高于 `fixed`；
6. 新增单元测试、学习文档、实验导航、双语 README 和进度记录；
7. 完整 `pytest -q` 测试套件通过。

## 非目标

- 不将排行榜解释为真实生产环境中的通用性能结论；
- 不实现随机任务生成、并发运行或机器学习训练；
- 不修改既有实验的实现或 `loop_engineering/` 公共接口。
